# Glossary

### Affected
Projects that have changed (directly or via transitive dependencies) since a given git ref. Used by `fx affected` to minimize work. See [Affected Analysis](/features/affected).

### Batch Execution
Combining multiple independent projects into a single process invocation (e.g., `dart test pkg_a pkg_b pkg_c`) to reduce process startup overhead. See [Batch Execution](/features/batch-execution).

### Cache Entry
A stored task result containing exit code, stdout, stderr, duration, input hash, and output artifacts. Stored as JSON in `.fx_cache/`.

### Cache Hit
When the computed input hash matches an existing cache entry, allowing the result to be replayed instantly without re-executing the task.

### Cache Miss
When the computed input hash has no matching cache entry, requiring the task to execute normally.

### Conformance Rule
A workspace-wide check enforcing architectural standards — e.g., all projects must have tests, names must follow conventions. See [Conformance Rules](/recipes/conformance-rules).

### DAG
Directed Acyclic Graph. The project dependency graph is a DAG, which guarantees a valid topological sort exists (no cycles).

### `dependsOn`
Target configuration that declares prerequisite targets. `dependsOn: [build]` means "run build before this target." The `^` prefix means "run on dependency projects." See [Task Pipeline Configuration](/concepts/task-pipeline-configuration).

### Executor
The command that implements a target. Can be a shell string (`dart test`) or a plugin reference (`plugin:custom_builder`). See [Executors](/concepts/executors).

### `.fxignore`
A file at the workspace root that excludes directories from project discovery, using gitignore-style patterns. See [.fxignore](/reference/fxignore).

### Generator
A scaffolding tool that creates new projects or files in the workspace from templates. See [Generate Code](/features/generate-code).

### Inferred Task
A target automatically detected from project file structure — e.g., a `test/` directory causes fx to infer a `test` target. See [Inferred Tasks](/concepts/inferred-tasks).

### Input Hash
A SHA-256 hash computed from all files and values that affect a task's outcome. Used as the cache key. See [How Caching Works](/concepts/how-caching-works).

### Module Boundary
A tag-based rule restricting which projects can depend on which others, enforced by `fx lint`. See [Enforce Module Boundaries](/features/enforce-module-boundaries).

### Named Configuration
An environment-specific override set (e.g., `ci`, `production`) activated with `--configuration`. See [Types of Configuration](/concepts/configuration).

### Named Input
A reusable set of input patterns referenced by name (e.g., `{testing}`). Defined in `namedInputs` and expanded recursively. See [Inputs and Named Inputs](/recipes/inputs-and-named-inputs).

### Output Artifact
A file generated by a target (e.g., `*.g.dart`) that is cached and restored on cache hits. Configured via the target's `outputs` patterns.

### Plugin
An extension point for customizing fx. Three types: executor plugins, generator plugins, and conformance rule handlers. See [Plugins](/concepts/plugins).

### Project
A single package in the workspace, discovered from its `pubspec.yaml` file. Each project has a name, type, path, tags, and targets.

### Project Graph
The DAG of workspace project dependencies, built from `pubspec.yaml` path dependencies. The foundation for task ordering, affected analysis, and boundary enforcement. See [Explore Your Workspace](/features/explore-your-workspace).

### Remote Cache
A shared cache store (HTTP, S3, GCS, or filesystem) that allows cached results to be reused across machines and CI runners. See [Remote Cache](/recipes/remote-cache).

### Root-Level Script
A workspace-level command defined in `fx.scripts`, invoked with `fx run :name`. Not tied to any project. See [Root-Level Scripts](/recipes/root-level-scripts).

### Tag
A label assigned to a project (in its `pubspec.yaml` under `fx.tags`) for module boundary matching and conformance rules. Tags support wildcard patterns.

### Target
A named task (e.g., `test`, `build`, `analyze`) with an executor, inputs, outputs, and dependencies. Targets can be inferred, defined at workspace level, or overridden per-project.

### Target Defaults
Lowest-priority target definitions (in `targetDefaults`), overridden by workspace targets and project targets.

### Task Graph
The graph of tasks (project + target combinations) and their dependencies, computed from the project graph and `dependsOn` configuration. Determines execution order.

### Task Pipeline
The resolved execution order for a target, including its `dependsOn` chain and transitive `^` dependencies. See [Task Pipeline Configuration](/concepts/task-pipeline-configuration).

### Topological Sort
Kahn's algorithm ordering projects so that dependencies always run before their dependents. Guarantees correct execution order in a DAG.

### Transitive Dependent
A project that indirectly depends on another through a chain of dependencies. If `app → models → shared`, then `app` is a transitive dependent of `shared`.

### Worker
A CI matrix agent assigned a deterministic partition of projects for distributed execution. See [Distribute Tasks](/features/distribute-tasks).

### Workspace
The root directory containing all managed projects, configured via `pubspec.yaml` or `fx.yaml`. Defines targets, boundaries, cache settings, and other workspace-wide configuration.
