# Inputs and Named Inputs

Input patterns control which files are included in the cache hash computation. Getting inputs right is critical to effective caching:

- **Too broad** — Unrelated file changes trigger cache misses, wasting time
- **Too narrow** — Relevant file changes are missed, serving stale results
- **Just right** — Cache invalidates exactly when the output would change

## Input Patterns

```yaml
fx:
  targets:
    test:
      inputs:
        - lib/**
        - test/**
        - pubspec.yaml
```

### Glob Patterns

| Pattern | Matches |
|---------|---------|
| `lib/**` | All files recursively under `lib/` |
| `*.dart` | Dart files in the project root only |
| `test/**/*.dart` | Dart files recursively under `test/` |
| `pubspec.yaml` | Exact file match |
| `{projectRoot}/lib/**` | Explicit project-relative glob |
| `{workspaceRoot}/.github/**` | Workspace-relative glob |

### Exclusion Patterns

Prefix with `!` to exclude files from matching:

```yaml
fx:
  targets:
    test:
      inputs:
        - lib/**
        - test/**
        - "!**/*.md"           # Markdown changes don't affect tests
        - "!**/.gitkeep"       # Ignore placeholder files
        - "!**/generated/**"   # Ignore generated code (it has its own target)
```

Exclusions are processed after inclusions — they remove files from the matched set.

### Special Patterns

| Pattern | Resolves To | Use Case |
|---------|-------------|----------|
| `env('DART_DEFINES')` | Value of the environment variable | Build-time config |
| `env('BUILD_MODE')` | Value of the environment variable | Dev vs. prod builds |
| `runtime('dart --version')` | Command output | SDK version tracking |
| `runtime('flutter --version')` | Command output | Flutter SDK tracking |
| `{externalDependencies}` | Contents of `pubspec.lock` | Dependency versions |

### Environment Variables in Hashes

Include environment variable values so cache entries differ between environments:

```yaml
fx:
  targets:
    build:
      inputs:
        - lib/**
        - env('DART_DEFINES')      # Different defines → different hash
        - env('BUILD_MODE')         # release vs. debug
        - env('API_BASE_URL')       # Different environments
```

If the variable is unset, it's hashed as an empty string. Changing the variable value invalidates the cache.

### Runtime Values in Hashes

Include command output for dynamic values that aren't files or environment variables:

```yaml
fx:
  targets:
    build:
      inputs:
        - lib/**
        - runtime('dart --version')       # SDK version
        - runtime('flutter --version')    # Flutter SDK version
        - runtime('git rev-parse HEAD')   # Current commit
```

Runtime commands are executed before hashing. Their stdout is included in the hash computation.

## Named Inputs

When multiple targets share the same input patterns, define reusable named sets:

```yaml
fx:
  namedInputs:
    default:
      - lib/**
      - pubspec.yaml
      - "{externalDependencies}"
    testing:
      - "{default}"            # References the 'default' set
      - test/**
      - test_fixtures/**
    building:
      - "{default}"
      - build.yaml
      - "!lib/**/*.g.dart"     # Exclude generated code
    production:
      - "{default}"
      - env('BUILD_MODE')
      - env('DART_DEFINES')
```

Reference named inputs in targets:

```yaml
fx:
  targets:
    test:
      inputs:
        - "{testing}"
    build:
      inputs:
        - "{building}"
    deploy:
      inputs:
        - "{production}"
```

### Expansion

Named inputs expand recursively:

```text
{testing}
  → {default}, test/**, test_fixtures/**
  → lib/**, pubspec.yaml, pubspec.lock contents, test/**, test_fixtures/**
```

You can also mix named inputs with additional patterns:

```yaml
fx:
  targets:
    integration-test:
      inputs:
        - "{testing}"
        - integration_test/**     # Additional patterns
        - env('TEST_API_URL')     # Additional env var
```

Circular references (e.g., `a` references `b`, `b` references `a`) are detected and reported as errors at startup.

## Output Patterns

Define which files are generated by a target:

```yaml
fx:
  targets:
    build:
      outputs:
        - "lib/**/*.g.dart"    # Generated code
        - "build/**"           # Build artifacts
        - "coverage/"          # Coverage reports
```

Output files are collected after execution and stored with the cache entry. On cache hit, they're restored to disk — making it appear as if the target actually ran.

<Info>
Without `outputs` configuration, only terminal output is replayed on cache hit. If your target generates files that other targets need, always configure `outputs`.
</Info>

## Common Patterns

### Dart Package Test

```yaml
test:
  inputs:
    - lib/**
    - test/**
    - pubspec.yaml
    - "{externalDependencies}"
    - "!**/*.md"
```

### Code Generation

```yaml
build:
  inputs:
    - lib/**
    - build.yaml
    - pubspec.yaml
    - "!lib/**/*.g.dart"      # Exclude generated files from input
  outputs:
    - lib/**/*.g.dart         # But include them in output
```

### Flutter Build

```yaml
build:
  inputs:
    - lib/**
    - assets/**
    - pubspec.yaml
    - env('DART_DEFINES')
  outputs:
    - build/app/outputs/**
```

## Troubleshooting

### Cache Misses When Nothing Changed

Your inputs may be too broad. Check if non-relevant files are being included:

```text
$ fx run core test --verbose
```

The verbose output shows which files are included in the hash. Look for files that shouldn't affect the target.

### Stale Cache Hits

Your inputs are too narrow — a relevant file isn't being tracked. Add the missing pattern to `inputs`.
