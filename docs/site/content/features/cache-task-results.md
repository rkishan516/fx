# Cache Task Results

fx provides a computation caching system that prevents tasks from running twice when inputs haven't changed. This accelerates local development and significantly reduces CI costs.

The caching system restores both **terminal output** (stdout/stderr) and **files generated** during task execution (such as build outputs or generated code).

## How Caching Looks

```text
$ fx run core test
> Running "dart test" for core...
> 00:03 +24: All tests passed!
core:test completed in 3.2s

$ fx run core test
> core:test — replayed from cache (0.01s)
```

The second run is **instant** — fx replays the cached terminal output without executing anything.

## Defining Cacheable Tasks

Configure which targets are cached in your workspace config:

```yaml
fx:
  targets:
    test:
      executor: dart test
      cache: true        # Enable caching (default)
      inputs:
        - lib/**
        - test/**
    build:
      executor: dart run build_runner build
      cache: true
      inputs:
        - lib/**
        - build.yaml
      outputs:
        - lib/**/*.g.dart
    deploy:
      executor: ./scripts/deploy.sh
      cache: false       # Never cache deploy
```

<Info>
Cacheable tasks must be **side-effect free** — given the same inputs, they should always produce the same outputs. Tasks like deployment, sending notifications, or calling external APIs should NOT be cached.
</Info>

## Fine-Tuning with Inputs and Outputs

Two controls manage caching behavior:

- **Inputs** — Files and values that affect the cache hash. If any input changes, the cache is invalidated.
- **Outputs** — Files generated by the task that should be restored on cache hit.

### Inputs

```yaml
fx:
  targets:
    build:
      inputs:
        - lib/**                    # All source files
        - pubspec.yaml              # Package manifest
        - "!lib/**/*.md"            # Exclude markdown files
        - "{externalDependencies}"  # pubspec.lock content
        - env('DART_DEFINES')       # Environment variable
```

Exclude files that don't affect the task outcome (like README changes):

```yaml
fx:
  targets:
    test:
      inputs:
        - lib/**
        - test/**
        - "!**/*.md"          # Markdown changes don't affect tests
        - "!**/.gitkeep"      # Ignore placeholder files
```

See [Inputs and Named Inputs](/recipes/inputs-and-named-inputs) for advanced patterns.

### Outputs

```yaml
fx:
  targets:
    build:
      outputs:
        - lib/**/*.g.dart          # Generated code
        - build/                   # Build artifacts
        - coverage/                # Coverage reports
```

When a cached build is restored, fx writes these output files back to disk — making it appear as if the build actually ran. Without `outputs` configuration, only terminal output is restored.

## Automatic Cache Configuration

fx automatically infers targets from your project structure, and inferred targets come with sensible cache defaults:

| Detected | Default Inputs |
|----------|---------------|
| `test/` exists | `lib/**`, `test/**` |
| `build.yaml` exists | `lib/**`, `build.yaml` |
| `analysis_options.yaml` exists | `lib/**` |

You only need explicit configuration when the defaults aren't right for your project. View what fx auto-configured:

```text
fx show <project-name> --targets
```

## What Gets Hashed

The input hash is a SHA-256 computed from:

| Component | Purpose |
|-----------|---------|
| Target name + executor | Different targets produce different hashes |
| Dart SDK version | Ensures cache validity across SDK updates |
| Matching source files | Sorted by path for cross-platform determinism |
| `pubspec.lock` content | Dependency version changes invalidate cache |
| Environment variables | When using `env('VAR')` patterns |
| Runtime command output | When using `runtime('cmd')` patterns |

Files are **sorted by path** before hashing, ensuring the same hash regardless of filesystem ordering.

## Remote Caching

By default, fx caches locally in `.fx_cache/`. For team-wide caching, configure a remote store so cached results are shared across developer machines and CI runners.

```yaml
fx:
  cache:
    enabled: true
    directory: .fx_cache
    remoteUrl: https://cache.example.com
```

A cache computed on one developer's machine can be reused by CI, and vice versa:

```text
# Developer A runs tests (cache miss, result stored remotely)
$ fx run core test
> core:test completed in 3.2s

# CI checks out same code (cache hit from remote)
$ fx run core test
> core:test — replayed from cache (0.05s)
```

fx supports HTTP, S3, GCS, and filesystem remote stores. See [Remote Cache](/recipes/remote-cache) for setup guides.

## Cache Management

```text
$ fx cache status
Cache location: .fx_cache/
Entries: 42
Total size: 12.3 MB

$ fx cache clear
Cleared 42 cache entries
```

## LRU Eviction

Prevent unbounded cache growth with `maxSize`:

```yaml
fx:
  cache:
    maxSize: 500    # MB
```

When the cache exceeds the limit, the oldest entries (by last-modified time) are evicted first.

## Skipping the Cache

### Per-Command

```text
fx run-many --target test --skip-cache
```

### Globally

```yaml
fx:
  skipCache: true
```

### Per-Target

```yaml
fx:
  targets:
    deploy:
      cache: false
```

## Troubleshooting Cache Issues

### Cache Misses When You Expect Hits

If tasks re-run when you expect a cache hit:

1. Check that `inputs` patterns cover all relevant files — a file change outside the pattern won't invalidate the cache, but a file change *inside* the pattern will
2. Ensure `pubspec.lock` hasn't changed (it's included by default via `{externalDependencies}`)
3. Check for environment variables that differ between runs
4. Run with `--verbose` to see the computed hash

### Cache Hits When You Expect Misses

If stale results are returned:

1. Your `inputs` patterns may be too narrow — the changed file isn't being tracked
2. Run `fx cache clear` to start fresh
3. Add the missing file pattern to `inputs`
